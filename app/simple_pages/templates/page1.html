GIT - WHAT IS IT?

According to git-scm.com, "Git is a free and open-source distributed version control system designed to handle everything from small to very large projects with speed and efficiency". An in-progress, or completed, project's code can be stored in a GitHub repository for proper management and version control of the source code. Git is regarded as the dominant source code management software (SCM) at this time. It was initially designed and developed by Linus Torvalds for Linux kernel development in 2005.

Version control systems, like Git, allow individuals or groups to work on a project while providing all of the tools necessary to ensure that no mishaps take place in the code that could interfere with someone else's work or break the entire project.

In the sections below I will go into detail about Git regarding:

  • Repositories
  • Branches
  • Tags
  • Commits
  • Merges

REPOSITORIES

A GitHub repository is the online home for the source code of a project. Any user can create a GitHub account, which will then allow them to make repositories on the GitHub website. These can be set to Public or Private view, as dictated by the user when making a new repository. The GeeksForGeeks website describes GitHub repositories as, "a collection of files of various different versions of a Project...A GIT repository allows performing various operations on it to create different versions of a project. These operations include the addition of files, creating new repositories, committing an action, deleting a repository, etc."

A repository is a project that is stored and worked off of by a team. The functions of the repository allow for proper version control and testing of code before it is considered finalized. Source code can be pulled down from a repository by a user and then updated on their own time. Repositories also allow source code to be safely updated by managing the uploads of new branches/work from team members in real time.

Repositories offer many different tools and functions through GitHub for testing and managing your source code. Integrating your code into a development or production environment, testing to ensure compatability with Master branch and other web apps, and scanning for any errors that may have been missed are all available via GitHub Repositories.

BRANCHES

Branches are a function of Git that allow the user to preserve the integrity of their project while updates are made. Branches function by tracking all changes to the source code Main or Master branch, also known as the "source of truth". The Master branch of the project is always treated as the most up-to-date version of the code from which everyone works from. Branches allow a team to break off into specific groups and work on different aspects of a project while ensuring that the entire project itself continues to function as intended.

For example: What if one person needed to create a new feature, such as a side navigation bar on an interior page, while another person needed to update hyperlink treatments on a separate interior page? Branches allow those team members to work simultaneously without jeopardizing the overall project code! The steps involved in working with branches are rather simple. Team members will have both local branches and remote branches from which they work from.

Local Branches

After a team member pulls down the latest Master branch from the repository that is the source of truth, they will have the latest and greatest code to work from on their own machine. This is the team member's Local Master branch. The team member will then access this code file and, either using console commands or a text editor, create a new branch to begin working from locally. This new local branch is separated from the Master branch, and it ensures that any updates made to the code won't interfere with the Master branch until it is ready to be merged (see section on Merge).

The Master branch that now lives on their machine, as well as the new branch that they created to work from, are both Local branches. They are not connected to the repository that the whole team has access to. Local branches are saved only to the team member's machine.

The team member can make as many local branches as they need to when making updates to the project. When they are done making changes in a local branch the team member can test the code as needed to ensure compatability as well as functionality with the Master branch. After this is completed with positive results, the local branch is then ready to be pushed to the remote branch, which lives in the remote repository.

Remote Branches

A Remote branch is a team member's local branch, of the same name, that has been pushed to the online GitHub repository. The updates that were made to the local branch will be visible as a new remote branch for the team to see and work with in the online repository. It will stay separated from the Master branch until it is reviewed and approved to merge.

Remote branches are extremely beneficial to the version control process because the Master branch remains safe even after a team member pushes their changes to the online repository. Code review and any further testing that is necessary can take place one final time in this stage. Once everything in the new Remote branch has been confirmed as correct, functional, and not in conflict with the Master branch, a merge can take place.

TAGS

Within Git there is a feature referred to as "tagging". Tags are not the same as branches or commits, although there does seem to be some confusion about this online. Tags function as specific reference points in a project history, often used to denote version of releases for a project. Another way to say this is that tags are symbolic names given to a revision that always point to the same object, and they do not change with commits.

The tutorials page on ATLASSIAN Bitbucket states, "Git tags are like branches that do not change...Tagging is an additional mechanism used to create a snap shot of a Git repository. Tagging is traditionally used to create semantic version number identifier tags that correspond to software release cycles." Git supports 2 different types of tags: Lightweight and Annotated. Lightweight tags store less metadata than Annotated tags and are generally less preferred than their counterparts.

COMMITS

Commits are essentially save points in a project that are made as work is being done. They are the changes to a code file, or set of files, that get saved within the branch that is being worked in at the time of commit. When a commit is made a unique ID is assigned to the change, or set of changes, that allows detailed records to be kept. In this way it is made clear what exact changes were incorporated to each file, at what time they were made, and who made them. Commits should always contain a commit message, which is a brief description of what changes were made by the user for everyone to review.

Good commits are essential to the ease of work for a team. A good commit not only means working on the correct file and branch for a project, but it also means that commit messages are clear and concise. Additionally, good commits should not be overly burdensome for the team. It is much easier to review consecutive, small changes than it is to review massive amounts of changes across multiple different files within a project.

MERGES

Merging in Git is when you join two or more development histories together into one branch of the project. Commits in the separate histories, or branches, will be incorporated into the Master branch most often when a merge request is completed, from my experience. Merging can take place within the GitHub repository for the project itself via simple interface options that will be presented upon a new remote branch upload.

Merges are a vital step in the development workflow for coding projects. They allow a team to safely work off of each other's updates on a project, up to the point of last merge. What does this mean? As each team member is locally-updating, testing, pushing, and then merging their work into the Master branch of a project on a remote repository, each team member can be assured that they are not working off of old code that may be missing latest features or updates. This is just another way to ensure that no individual changes to files interrupt or break the entire project as everyone should be working off of the latest files.
